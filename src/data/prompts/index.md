## Prompts

### ask-me

```md
私のメッセージの内容のあらゆる側面について、私たちが共通の理解に達するまで、徹底的に私に質問してください。
議論の対象の選択肢を順にたどり、決定事項の間の依存関係を一つずつ解決してください。
各質問に対して、必ずあなたの推奨する回答を提示してください。
質問は一度に一つずつ行ってください。
あなたの知識で判断できる項目については、私に質問する代わりに、判断とその根拠を一言で述べてから、次の質問に進んでください。
ただし、趣味嗜好や方針に関わること、誤っていた場合に議論を大きく巻き戻す必要があることは、必ず私に質問してください。
それでは、以下の私のメッセージに基づいて、質問を始めてください。
```

### understanding

```md
## 指示

会話を通じて、私が取り上げる対象を理解する手伝いをしてください。
私が対象の構造や関係を追い、その理解を必要に応じて予測、説明、判断に使えるように、
理解にかかる認知負荷を制御してください。

### 進め方

私の質問や目的に応じて、次の流れを繰り返してください。

1. 私が今、何を理解しようとしているかを読み取る。
2. 扱う内容を一つに絞る。
3. 扱う内容の位置づけと関係を説明する。
4. 私が次に扱ってほしいことを返せるように問いかける。
5. 私の返答を待ち、それに応じて、補足するか、扱う内容の細かさを変えるか、次の内容へ進む。

問いかけたら、そこで返信を終えてください。
私が途中で質問や関心を変えた場合は、新しい目的に合わせて次に扱う内容を選び直してください。

### 内容の選び方と説明

一度に扱う内容は、私が次の発言で「追えたところ」と「詰まったところ」を返せる大きさにしてください。

対象について、まとまりのある内容を一つ選んでください。
たとえば、役割や責務、情報や値の流れ、条件とそれに応じた判断、
状態や状況の変化、主張とその根拠、要素同士の関係などです。
その内容を追うために密接に関係する情報は、まとめて扱って構いません。

今扱っている内容の位置づけを、理解に必要な範囲で短く示してください。
詳細は、その内容に含まれる関係を追うために必要なものに絞ってください。

説明する順序は、対象の構造や関係が理解しやすくなるように決めてください。
元の順序が関係の理解に役立つ場合は、その順序を活かしてください。
今扱っている内容に直接関係しない細部、例外、周辺的な情報は後に回してください。

### 問いかけ

説明の後には、次のターンで扱う焦点を私から受け取るために、
私が次に扱ってほしいことを自分の言葉で一つ返せるように問いかけてください。
私の理解状態や次の関心を先回りして仮定せず、返答の内容は私に委ねてください。

一度に行う問いかけは一つとし、その問いで私が答える対象も一つに絞ってください。

私が理解しようとしていることが、これまでの会話や提示された情報から読み取れる場合は、
その目的に基づいて進めてください。
目的によって扱う範囲や説明方針が大きく変わる場合だけ、説明を始める前に問いを一つ出してください。

### 根拠と不確実性の扱い

会話の中では、情報を次のように区別して扱ってください。

- 提示された情報から直接確認できること
- 提示された情報をもとに推定したこと
- 一般的な知識や慣習から補ったこと
- 現時点では判断できないこと

根拠を示す際は、対象に応じて、資料の名前、見出しなど、
該当箇所を特定できる情報を可能な範囲で示してください。
不確実性が説明に影響する場合は、その不確実性を明示してください。

---

それでは、以下の私のメッセージに基づいて、私が取り上げる対象を理解する手伝いをしてください。
```

###　decision-distillation

```md
# 目的

これまでの議論と参照情報を読み、現時点で採用されている内容と、
次に決める必要がある論点を整理してください。
今の議論で有効な前提、決定、帰結、残る論点だけが分かる形で整理してください。
議論の履歴要約にはしないでください。
矛盾を検出した場合は、整理結果を出力する前に、解消に必要な一点をユーザーへ確認してください。

# 根拠を確認する

これまでの議論を読んでください。
判断に関係するコード、ドキュメント、参照されたWebページを読んでください。
判断に関係する内容は、参照できる根拠に当たって確認してください。
参照できる根拠がない内容、現在のツールでは確かめられない内容、
根拠が曖昧な内容は、未確認として扱ってください。

# 分類する

前提、決定事項、必然的な帰結、残る論点を分けてください。

前提には、議論の前から所与だった条件、ユーザーが前提として示した内容、
コードやドキュメントから確認できた事実を書いてください。
決定事項には、議論の中でユーザーが明示的に採用した判断を書いてください。
必然的な帰結には、前提と決定事項から論理的に外せない内容を書いてください。
残る論点には、ユーザーの目的に沿って作業を進めるために、
これから実際に決める必要がある事項を書いてください。

仮定、推測、曖昧な点、情報不足は、それが次の判断を左右する場合だけ残る論点に書いてください。
非採用、対象外、保留、未確認の情報は、それ自体を採用済みの決定として書かないでください。
ただし、ユーザーが明示的に「採用しない」「今回は対象外」「保留」と決めたことは、
現在の判断範囲を誤らせる恐れがある場合に限り、
否定形の決定または範囲指定として決定事項に書いてください。
確認できない外部情報やアシスタントの推測は、確認済みの前提に混ぜないでください。

# 案の扱い

アシスタントの提案、実装案、推測は、ユーザーが採用した時点で決定事項として扱ってください。
議論の途中で出た案は、最新のユーザー発言と採用済みの判断に照らして扱いを決めてください。

採用済みの案は、決定事項に書いてください。
次に実際に決める必要がある案は、残る論点に書いてください。
採用されていない案、最新の方針から外れた案、単に提示されただけの案は、
決定事項にも残る論点にも書かないでください。

# 矛盾の判定

矛盾は、同じ前提のもとで同時に成立しない、確認済みの内容同士に限って判定してください。
後の発言が以前の内容を明示的に修正している場合は、最新の内容に更新してください。
未決事項、追加検討、情報不足、表現の違い、提示されただけの案は、矛盾として扱わないでください。
曖昧さや情報不足は、それが次の判断を左右する場合だけ残る論点に書いてください。

# 質問

質問してよいのは、整理を進める前に解消すべき矛盾だけです。
矛盾を検出した場合は、解消に必要な質問を一つだけしてください。
複数の矛盾を検出した場合は、他の矛盾の判定を左右するものを一つ選んでください。
質問には、確認済みの内容同士の食い違いと、ユーザーに決めてもらう一点を短く含めてください。
質問する場合は、その質問だけを出力してください。
解消していない矛盾は、そのまま残してください。
確定した内容は、確認済みの内容とユーザーの回答に基づいて更新してください。
ユーザーの回答を反映してから、残る矛盾をもう一度判定してください。

# 出力

矛盾がない場合、またはユーザーの回答によって矛盾が解消した場合は、
これまでの議論を次の四項目で簡潔に整理してください。

1. 前提
2. 決定事項
3. 必然的な帰結
4. 残る論点

各項目は、該当する内容だけで構成してください。
四項目を無理に埋めることより、分類の正確さを優先してください。
```
